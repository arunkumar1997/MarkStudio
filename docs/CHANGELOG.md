# Changelog

All notable changes to MarkStudio are documented here.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

Every user-facing change must add an entry under `Unreleased`. On release, the `Unreleased` section is renamed to the new version with a date (see [RELEASE.md](RELEASE.md)).

## [Unreleased]

### Added
* **Linting and formatting (T-121).** MarkStudio now has a lint/format gate. **ESLint** (flat config, `eslint.config.mjs`, ESLint 9 + `typescript-eslint`) encodes the lint-able rules from [CODING_GUIDELINES.md](CODING_GUIDELINES.md): `@typescript-eslint` recommended plus `eqeqeq`, no stray `console` (only `warn`/`error` are allowed in source; build scripts and test harnesses are exempt), explicit exported types (`explicit-module-boundary-types`), and `no-unused-vars` with a leading-underscore escape hatch. Build output and downloaded test runtimes are ignored; the CommonJS esbuild scripts at the repo root lint as Node/CommonJS, and test harnesses/mocks relax the exported-type and console rules. **Prettier** (`.prettierrc.json`) owns formatting — 2-space indent, double quotes, semicolons, no trailing commas, 80-column width, `endOfLine: auto` — with `eslint-config-prettier` applied last so the two never conflict; `docs/`, `*.md`, and the lockfile are in `.prettierignore`. New scripts: `npm run lint` (`eslint . --max-warnings 0 && prettier --check .`) and `npm run lint:fix`. A `lint` step joins the CI **build-and-test** job, so a lint or format violation now blocks merge ([TESTING.md](TESTING.md) §7). The existing tree was normalised in one pass (chiefly 4-space → 2-space indentation in several files); no behavioural change. **Five new dev dependencies** (`eslint`, `@eslint/js`, `typescript-eslint`, `prettier`, `eslint-config-prettier`) — dev-only, no runtime dependency or bundle-size change. *(2026-06-27)*
* **Continuous integration pipeline (T-120).** A GitHub Actions workflow (`.github/workflows/ci.yml`) now runs on every push to `main` and every pull request, and a red pipeline blocks merge ([TESTING.md](TESTING.md) §7). It has two jobs on `ubuntu-latest`. **build-and-test** runs `npm ci`, `npm run lint`, `npm run typecheck`, `npm run build`, then `npm test` (the unit + integration layers). **extension-host-tests** runs `npm ci` then `xvfb-run -a npm run test:exthost`: the Extension Host layer boots a real VS Code via `@vscode/test-electron`, which needs a display, so it runs under `xvfb` on the headless Linux runner; the downloaded VS Code build (~280 MB) is cached under `.vscode-test/` keyed on `package-lock.json`, so only the first run (or a dependency bump) re-downloads it. The Extension Host job is split out so a slow VS Code download never blocks the fast unit/integration feedback, and `concurrency` cancels superseded runs on the same ref. No app-code, dependency, or bundle-size change. *(2026-06-27)*
* **Extension Host lifecycle tests (T-113b).** MarkStudio now has the Extension Host test layer ADR-0012 deferred: the genuine Custom Editor lifecycle that neither the unit layer (T-112, mocked `vscode`) nor the jsdom integration layer (T-113, no running VS Code) can reach. A launcher (`test/exthost/runTests.ts`) runs in a plain Node process and boots a real VS Code via `@vscode/test-electron`, pointing it at this repository as the extension under development and at the bundled in-host suite (`dist-test/exthost-suite.cjs`) as the `extensionTestsPath`, opening a fresh temp workspace + user-data dir (`--disable-extensions`, `--disable-workspace-trust`) so the run is hermetic. The in-host suite (`test/exthost/index.ts` → `run()`) runs on a **minimal hand-rolled registry/runner** over `node:assert` (`test/exthost/harness.ts`) — **no Mocha/glob**, consistent with ADR-0005 / ADR-0011. A new `esbuild.exthost.js` builds two bundles with different externals: the launcher keeps `@vscode/test-electron` external (it downloads/spawns VS Code via dynamic requires), and the in-host suite keeps `vscode` external (the host injects it) and is **never** aliased to the mock, unlike the unit / integration builds. Because a custom editor's webview runs in an isolated iframe with no public introspection API, the first lifecycle tests assert the **host-observable** contract: the extension is discoverable and activates, `vscode.openWith` resolves the custom editor without error, an `edit` (a `WorkspaceEdit` — the same host path the webview's `edit` message takes) makes the document dirty and `save` clears it, and revert restores the on-disk content. `npm run test:exthost` builds → bundles → boots VS Code; it stays **out** of the default `test` (a full VS Code download is far heavier than the millisecond unit/integration runs), ready for CI to opt in (T-120). **One new dev dependency** (`@vscode/test-electron`). New **ADR-0013** records the harness design; the unit build (`esbuild.test.js`) now skips `test/exthost/` as well as `test/integration/`. No app-code or bundle-size change.
* **Integration-test harness and first DOM-seam tests (T-113).** MarkStudio now has an integration layer covering the webview seams the unit layer (T-112) could not reach because they need a real DOM. A new `esbuild.integration.js` mirrors the unit build (ADR-0011) — it discovers `test/integration/**/*.test.ts`, bundles them into `dist-test/integration.cjs`, and aliases the host-only `vscode` import to the mock — but imports `test/integration/_setup/dom.ts` **first**, which installs jsdom globals (`window`, `document`, `requestAnimationFrame`, MutationObserver, a no-op `ResizeObserver`, …) before any source module — and therefore CodeMirror — is evaluated. `npm run test:integration` builds and runs it on Node's built-in runner (`node:test`); `npm test` now runs type-check + unit + integration. The first 8 integration tests assert the genuinely DOM-bound guarantees: the CM6 `RemoteSync` echo path (`createEditor` — host `setContentFromHost` updates the document but must **not** round-trip back as a local `edit`), the minimal-diff cursor preservation when host content changes a region after the cursor (T-110), a user edit forwarding to `onLocalChange` as a diff batch, and the markdown-it preview's **incremental DOM patching** (`createPreviewRenderer` — an edit replaces only the changed block's `Element` while unchanged blocks keep their exact node identity, plus the 1-based source-line mapping scroll sync depends on and update coalescing). **One new dev dependency** (`jsdom` + `@types/jsdom`) — no new test framework; the runner stays `node:test` (ADR-0005, ADR-0011). New **ADR-0012** records the harness design and splits the genuine Extension Host lifecycle (custom-editor resolve / save / revert on `@vscode/test-electron`) into a follow-up (T-113b). No app-code or bundle-size change.
 MarkStudio now has an automated unit-test layer. `npm test` type-checks the tests (`tsconfig.test.json`), bundles every `test/**/*.test.ts` into `dist-test/tests.cjs` with esbuild — aliasing the host-provided `vscode` import to a small deterministic mock (`test/_mocks/vscode.ts`) — and runs the suite on Node's built-in test runner (`node --test`); `npm run test:unit` is the fast path without the type-check. **No new dependency** is added: the runner is `node:test` / `node:assert` and the bundler is the esbuild already used for the app (ADR-0011). First tests (43, all green) cover the MessageBus boundary guards in `src/messaging/messages.ts` (including the new `isMarkStudioConfig`-backed `init` / `configChanged` cases), the pure view-state logic in `src/webview/state/viewState.ts`, `MarkStudioDocument.applyChanges` (offset → `vscode.Range` mapping and single-`WorkspaceEdit` batching, via the mock), and `ConfigurationService` (`read` defaults / overrides and the `markstudio.*`-scoped `onDidChange`). New **ADR-0011** records the harness design; the CM6 `RemoteSync` echo path and the full Custom Editor lifecycle are deferred to an integration layer (T-113). No app-code or bundle-size change. MarkStudio now reads `markstudio.*` settings reactively and applies them to the editor live, without a reload. The first setting, **`markstudio.editor.lineNumbers`** (boolean, default `true`, `resource` scope), toggles the CodeMirror 6 line-number gutter. New host-side `src/services/ConfigurationService.ts` is a thin, stateless reader over VS Code's Configuration API: `read(resource?)` returns a plain-JSON `MarkStudioConfig` from `workspace.getConfiguration`, and `onDidChange` wraps `onDidChangeConfiguration` filtered to the `markstudio` section. The provider sends the resolved config on `init` (so the first paint already reflects it) and pushes a new `configChanged { config }` host → webview message on every change, re-reading per document URI so resource-scoped overrides win. Both `init.config` and `configChanged` are validated by a new `isMarkStudioConfig` boundary guard in `src/messaging/messages.ts` and documented in [api/message-protocol.md](api/message-protocol.md). The webview applies changes through a CodeMirror **`Compartment`** (`src/webview/editor/extensions.ts` exports `lineNumbersCompartment` + `lineNumbersExtension`; `createEditor.setConfig` dispatches `compartment.reconfigure(...)`), so the single long-lived `EditorView` is reconfigured — never rebuilt (ADR-0002). New **ADR-0010** records the stateless-reader + compartment design and the extensible settings seam. No new dependency; production-minified webview **~699.6 KB** (≈ unchanged over T-2.1), host bundle **~7.0 KB** (+~0.4 KB).
* **Editor ⇄ preview scroll synchronisation (T-2.1).** In `split` mode, scrolling either pane now scrolls the other to the matching Markdown block. `src/webview/preview/PreviewRenderer.ts` records each rendered block's 1-based source line (from the markdown-it token map) and exposes it through a new `getBlocks()`; because an edit above a block shifts its source line even when the rendered HTML is unchanged, the mapping is refreshed on every render rather than only on the blocks the HTML diff touches. The new `src/webview/preview/scrollSync.ts` maps the editor's top visible source line to a preview `scrollTop` (and back) by interpolating between the two block anchors that bracket it — CodeMirror's `lineBlockAtHeight` / `lineBlockAt` convert a source line to / from a vertical pixel offset, and the preview anchors come from the block nodes' positions within the scroll container. Both directions binary-search their anchor list so each scroll event reads layout O(log n) times. Feedback is suppressed per direction (each programmatic scroll arms a one-shot guard that swallows exactly its own echo event, and only when the scroll value actually changes), and the handlers no-op unless split mode is active so `editor-only` / `preview-only` cost nothing. Production-minified webview: **~699 KB** (+~2 KB over T-110); host bundle unchanged at **6.6 KB**. No new dependency, no protocol change.
* **External file-change reconciliation (T-110).** External edits to an open `.md` file (a revert, a `git pull`, a save from another editor, an external formatter) now update the editor without throwing the cursor to the top of the document. Because MarkStudio is a `CustomTextEditorProvider`, VS Code already keeps the `TextDocument` in sync with disk and fires `onDidChangeTextDocument`; the provider's existing (echo-guarded) handler already forwards that change to the webview as `setContent`. The fix is in `src/webview/editor/createEditor.ts`: `setContentFromHost` now reconciles the incoming text as a **minimal single-span diff** (trim the common prefix + common suffix via the new module-private `computeMinimalChange`) instead of replacing the whole document, so CodeMirror maps the existing selection across the change and the cursor stays put whenever the external edit is elsewhere in the file. No separate `FileSystemWatcher` is added — for a text-backed custom editor, `onDidChangeTextDocument` *is* VS Code's external-change reconciliation surface, and a raw watcher would double-fire on clean documents and risk clobbering unsaved edits on dirty ones. New **ADR-0009** records the decision and refines ADR-0004's generic "file watching" clause for the text-backed case. No new dependency, no protocol change; bundle sizes unchanged from T-109.
* **View-state and layout persistence (T-109).** CodeMirror cursor + scroll position and the App Shell's layout mode now survive a full extension reload (not just a tab switch). The webview view-state slot (`src/webview/state/viewState.ts`) gains a `cursor: { anchor, head } | null` and a `scrollTop: number` field; both are read on first `init` and replayed into the freshly built `EditorView` (selection through `EditorState.create({ selection })`, scroll through `view.scrollDOM.scrollTop = X` deferred to the next `requestAnimationFrame` so CM6's layout has run). The editor emits a debounced snapshot (250 ms) whenever the selection changes, the doc changes, or the scroll position moves, and `viewState.patch` writes it back through `vscode.setState()`. The new `src/services/StateStore.ts` is a thin `Memento` wrapper around `context.workspaceState` that stores the last layout mode keyed by document URI (`markstudio.layoutMode:${uri}`); on `init` the host reads it and includes it as the new optional `initialLayoutMode` field on the `init` message. The Memento value is authoritative — when present, `AppShell` overrides the value cached in `vscode.setState()` so the per-file layout preference survives reloads, not just tab switches. The webview notifies the host of every user-driven mode change via a new `layoutModeChanged` message (added to `src/messaging/messages.ts` with a boundary guard and documented in [api/message-protocol.md](api/message-protocol.md)); the host persists it through `StateStore.setLayoutMode`. `createEditor` was refactored to take an options object (`{ parent, initialText, initialCursor, initialScrollTop, onLocalChange, onSnapshotChange }`) so the new restore + snapshot seams have a clear place to live without a parameter explosion. Production-minified webview: **~697 KB** (+~2 KB over T-108); host bundle: **6.6 KB** (+~0.6 KB).
* **Core commands and keybindings (T-108).** Five new commands ship in `package.json` and `src/commands/registerCommands.ts`: `MarkStudio: Open in MarkStudio` (`markstudio.openMarkStudio` — reopens the active `.md` file in the MarkStudio editor via `vscode.openWith`), `MarkStudio: Toggle Preview` (`markstudio.togglePreview` — split ↔ editor-only, matching the built-in `markdown.showPreview` semantics), `MarkStudio: Toggle Split View` (`markstudio.toggleSplit` — split ↔ editor-only), `MarkStudio: Focus Editor` (`markstudio.focusEditor`), and `MarkStudio: Focus Preview` (`markstudio.focusPreview`; promotes the layout to `split` when invoked from `editor-only` so focus has somewhere to land). Default keybindings — `Ctrl+K V` (toggle preview), `Ctrl+K Ctrl+V` (toggle split), `Ctrl+K Ctrl+E` (focus editor), `Ctrl+K Ctrl+R` (focus preview) — are gated by `when: "activeCustomEditorId == 'markstudio.editor'"` so they activate only inside MarkStudio. Three new host → webview messages (`togglePreview`, `toggleSplit`, `focusPane`) are added to `src/messaging/messages.ts` with boundary guards and documented in [api/message-protocol.md](api/message-protocol.md); the toggles compute the target mode **inside** the webview, which owns the authoritative `vscode.getState()` value, so no host-side mode cache is introduced. `MarkStudioEditorController` gains `togglePreview()` / `toggleSplit()` / `focusPane()` next to the existing `setLayoutMode()`; `registerCommands.ts` centralises every command ID in a new `COMMAND_IDS` record next to `LAYOUT_COMMAND_IDS`. The webview wires `focusPane: "editor"` to CM6's `view.focus()` and `focusPane: "preview"` to the preview pane element (now `tabindex="-1"`) with a `:focus` outline keyed to `--vscode-focusBorder`. Production-minified webview: **~695 KB** (no measurable delta over T-107); host bundle: **6.0 KB** (+~1 KB).
* **Layout toolbar with Codicons (T-107).** The App Shell now mounts a small toolbar above the editor/preview panes (`src/webview/app/Toolbar.ts`) with three Codicon buttons — **Show Editor Only** (`codicon-edit`), **Show Editor and Preview** (`codicon-split-horizontal`), **Show Preview Only** (`codicon-preview`). Clicking a button switches layout mode through the same `AppShell.setLayoutMode` path as the command-palette commands; the active button reflects the current mode via `aria-pressed` and a `--vscode-toolbar-activeBackground` highlight. Buttons are keyboard-focusable and themed entirely through `--vscode-*` variables (ADR-0004, ADR-0005). The Codicons font + stylesheet are copied from `@vscode/codicons` into `dist/codicons/` by `esbuild.js` and loaded into the webview via `asWebviewUri` from `src/editor/webviewHtml.ts`; the existing `font-src ${webview.cspSource}` CSP rule already permits the `.ttf` fetch. `@vscode/codicons` is a `devDependency` — no runtime code dependency is added (production-minified webview: **~694 KB**, +~3 KB over T-106; the font is a separate 126 KB asset loaded on demand).
* **Resizable split with remembered layout mode (T-106).** The two long-lived panes now live inside a new App Shell (`src/webview/app/AppShell.ts`) that owns a draggable gutter (`src/webview/app/SplitView.ts`) and the three layout modes: `split` (default), `editor-only`, `preview-only`. Switching modes only toggles pane visibility / `flex-grow` via a `data-layout-mode` attribute on `#markstudio-root` — CodeMirror and the markdown-it preview are **never** unmounted or rebuilt. The gutter uses pointer events with `setPointerCapture` (correct touch/pen behaviour, drags do not break at the iframe edge), clamps each pane to a 160 px minimum, throttles drag updates to `requestAnimationFrame`, and resets to 50/50 on double-click. Per-webview view state (`splitRatio`, `layoutMode`) is read and written by the new `src/webview/state/viewState.ts` against `vscode.setState()` / `getState()` — defaults-fallback parsing tolerates partial or malformed state and remains forward-compatible for T-109's cursor/scroll snapshot.
* **Layout commands (T-106).** Three new commands are contributed in `package.json` and routed through `src/commands/registerCommands.ts`: `MarkStudio: Show Editor and Preview` (`markstudio.layout.showSplit`), `MarkStudio: Show Editor Only` (`markstudio.layout.showEditorOnly`), `MarkStudio: Show Preview Only` (`markstudio.layout.showPreviewOnly`). Each posts a new `setLayoutMode` host → webview message — added to `src/messaging/messages.ts` with a boundary guard and documented in [api/message-protocol.md](api/message-protocol.md). `MarkStudioEditorProvider` now tracks the active MarkStudio webview via `onDidChangeViewState` and exposes a `MarkStudioEditorController` so commands target the focused editor; if none is active, the command surfaces a clear status message instead of failing silently. The toolbar (T-107) and the wrapping toggle commands (T-108) will reuse these layout commands.
* **Live Markdown preview with incremental DOM patching (T-105).** The webview now hosts a second long-lived pane next to the CodeMirror 6 editor: a markdown-it preview that **patches** only the changed top-level Markdown blocks on every update (heading, paragraph, list, blockquote, code block, …). `src/webview/preview/PreviewRenderer.ts` builds one persistent `<article>` root, parses the document into markdown-it tokens, groups them by top-level block, renders each group to HTML, and diffs against the cached `{ html, node }[]` from the previous render using a longest-common-prefix + longest-common-suffix algorithm. Unchanged blocks keep their existing DOM `Element` reference, scroll position, and accessibility tree across keystrokes. Renders are debounced (40 ms) so bursts of keystrokes coalesce. markdown-it is configured with `html: false` for safety; raw HTML is escaped. The preview styling is keyed entirely to `--vscode-*` variables, so light, dark, and high-contrast themes are correct without a webview reload. ADR-0008 records the dependency choice, the rendering strategy, and the bundle-size impact (production-minified webview: **~687 KB**, up ~151 KB from T-104).
* **CodeMirror 6 source editor (T-104).** The webview now mounts a single, long-lived CM6 `EditorView` instead of the temporary `<textarea>`. `src/webview/editor/extensions.ts` configures the editor with the Markdown grammar (`@codemirror/lang-markdown`), undo/redo history, find-as-you-type search, multi-cursor + rectangular selection, line numbers, line wrapping, bracket matching, and a small `HighlightStyle` for headings/emphasis/strong/links/code/quotes. Every color, font, cursor, and selection token in the theme resolves to a `--vscode-*` variable, so light, dark, and high-contrast themes are handled correctly without recreating the editor. `src/webview/editor/createEditor.ts` exposes a tight surface (`setContentFromHost`, `focus`, `destroy`) and uses a `RemoteSync` `Annotation` to distinguish host-applied transactions from user edits so they never round-trip. New ADR-0007 records the seven `@codemirror/*` / `@lezer/highlight` packages introduced (production-minified webview bundle: ~536 KB).
* **Diff-based `edit` messages.** The webview ⇄ host `edit` payload now carries the CodeMirror diff as `changes: EditChange[]` plus the resulting `text` (for the host's echo guard). On the host side, `MarkStudioDocument.applyChanges` translates each change's offset range into a `vscode.Range` and routes the batch through a single `vscode.WorkspaceEdit`, so multi-cursor edits and large edits remain a single undo step. `replaceAll` is retained for future reconciliation paths (T-110). The boundary guard in `src/messaging/messages.ts` and the protocol reference at [docs/api/message-protocol.md](api/message-protocol.md) are updated together.
* **Typed MessageBus (T-103).** All host ⇄ webview traffic now flows through a typed, discriminated-union message contract. `src/messaging/messages.ts` defines the `init`, `setContent`, `error` (host → webview) and `ready`, `edit`, `error` (webview → host) shapes and ships boundary guards (`isHostToWebviewMessage`, `isWebviewToHostMessage`) that reject any payload that does not match. `src/messaging/HostMessageBus.ts` and `src/messaging/WebviewMessageBus.ts` are thin per-side wrappers — the split exists because the host bundle imports `vscode` and the webview bundle does not, so a single file would force one bundle to pull in the other's runtime. `MarkStudioEditorProvider` and the webview entry route every message through their respective bus; malformed messages are dropped and logged instead of crashing the handler. The host→webview `update` shim from T-102 has been renamed `setContent` to match the contract documented in [ARCHITECTURE.md](ARCHITECTURE.md) §6, and the protocol now has a stable reference at [docs/api/message-protocol.md](api/message-protocol.md).
* **Document model and two-way editing (T-102).** New `src/editor/MarkStudioDocument.ts` wraps `vscode.TextDocument` and funnels every write through `vscode.WorkspaceEdit`, so dirty state, undo/redo, save and revert integrate with VS Code natively (ADR-0001). `MarkStudioEditorProvider` now routes webview `edit { text }` messages through `MarkStudioDocument.replaceAll`, and pushes `setContent { text }` back to the webview only on **external** changes (revert, on-disk edits, other editors) — guarded by a text-equality echo check so the webview never receives a copy of its own edit. The webview scaffold (`src/webview/main.ts`) now uses a temporary `<textarea>` (replaced by CodeMirror in T-104) that emits `edit { text }` on input and reconciles host `setContent`s while clamping the caret, making the round-trip demonstrable end-to-end.
* **Extension scaffold (T-101).** The repository is now a buildable VS Code extension: `package.json` manifest with the `customEditors` contribution point registering **MarkStudio** as an `option` editor for `*.md`; strict `tsconfig.json`; an `esbuild.js` build script producing dual bundles (`dist/extension.js` for the Node host, `dist/webview.js` for the browser webview) with a `--watch` mode; `.vscodeignore` and `.gitignore`.
* **VS Code debug wiring** under `.vscode/`: `launch.json` (an `extensionHost` "Run MarkStudio Extension" config so **F5** launches an Extension Development Host) with a `npm: build` `preLaunchTask`, `tasks.json` (build + esbuild watch tasks), and `extensions.json` recommendations.
* **Activation entry point** `src/extension.ts` registering a minimal `MarkStudioEditorProvider` (`src/editor/MarkStudioEditorProvider.ts`) — one retained webview per editor instance (`retainContextWhenHidden`), `webview.html` assigned once via `src/editor/webviewHtml.ts` with a strict CSP + per-load nonce, and a placeholder webview runtime (`src/webview/main.ts`) that renders the document text using `--vscode-*` theme variables.
* Documentation foundation under `docs/`: `README`, `PROJECT_STATUS`, `AGENT_HANDOFF`, `ARCHITECTURE`, `DECISIONS`, `ROADMAP`, `FEATURES`, `TODO`, `CHANGELOG`, `CONTRIBUTING`, `TESTING`, `RELEASE`, `CODING_GUIDELINES`.
* Working subdirectories with index READMEs: `docs/design/`, `docs/implementation/`, `docs/api/`, `docs/research/`, `docs/screenshots/`.
* Foundational Architecture Decision Records ADR-0001 … ADR-0006 capturing the Custom Editor API, the single persistent webview, the CodeMirror 6 / markdown-it choice, reliance on VS Code for theming/autosave/file-watching, the no-framework stance, and the esbuild bundler.
* GitHub contribution scaffolding referenced by the spec structure: `.github/PULL_REQUEST_TEMPLATE.md` and `.github/ISSUE_TEMPLATE/` (bug report, feature request, config).

### Changed
* **`MarkStudioEditorProvider.register`** now returns `{ provider, disposable }` instead of a bare `vscode.Disposable`, so `src/extension.ts` can hand the provider to `registerCommands` for active-editor targeting (T-106). The disposable still owns the custom-editor registration; the provider also tracks the active webview via `onDidChangeViewState` and clears the reference on disposal.
* **ADR-0006 (Bundle with esbuild)** moved from `Proposed` to `Accepted` after T-101 confirmed the dual-target build.
* **`MarkStudioEditorProvider`** routes webview `edit` through `MarkStudioDocument.applyChanges` (T-104) instead of `replaceAll` (T-102), so user edits land as targeted ranges in the undo stack rather than full-document replacements.
* Documentation consistency pass: reconciled the Phase 0 vs Phase 1 labeling of the scaffolding tasks (T-101…T-103 are Phase 0 / milestone M0.3) in `TODO.md`; clarified that there is **one persistent webview per editor instance** (a single shell hosting both panes) across `README`, `ARCHITECTURE`, and `TODO`; aligned the workflow loop and required-reading list in `.github/copilot-instructions.md` with `WORKFLOW.md`; added the `.ai/` directory to the Project Structure in `PROJECT_SPEC.md`; marked roadmap milestone M0.2 done.

### Notes
* The custom editor is registered with `priority: "option"`, so `.md` files keep their default editor and MarkStudio is available via **Reopen Editor With…** or the new **MarkStudio: Open in MarkStudio** command (an intentionally non-surprising default; revisit before release). With T-110 the Phase 1 editing-core robustness story is complete: cursor + scroll + per-file layout mode all survive a full reload (T-109), and external on-disk changes reconcile through the managed `TextDocument` with a cursor-preserving diff (T-110, ADR-0009) — Phase 1 milestones M1.1, M1.2, M1.3, M1.4, M1.6 and M1.7 are closed. With T-2.1 the editor and preview panes now scroll together in split mode, closing the most visible remaining Phase 1 UX gap. With T-111 the first reactive `markstudio.*` setting (`editor.lineNumbers`) toggles live via a CM6 `Compartment` (ADR-0010), establishing the settings seam future options extend. The first test harness (T-112) is next.
